## Architectural decisions (worth thinking about explicitly)

### Decision 1: One Docker container vs multiple

**Recommendation: One container, one TradingNode, multiple strategies.**

Pros (one container):
- Single source of truth for PRM + Allocator
- No IPC for shared risk state
- Single graceful shutdown / kill switch
- Single IBKR API connection (avoids API key juggling + simultaneous-login issues)
- Operationally simpler

Cons (one container):
- One bad strategy can crash the node (mitigation: per-strategy try/except in `on_bar`)
- Less fault isolation between strategies

The alternative (multi-container) has substantial operational complexity:
shared PRM state requires file-system synchronization or a separate
state-coordination service; IB connections multiply; kill-switch
coordination is hard. **Not recommended at this AUM scale.**

### Decision 2: Allocator type

**Current:** inverse-vol with correlation penalty in `PortfolioAllocator`.

**V3.7 added:** ERC (Equal Risk Contribution) as a research module
(`titan/research/framework/allocator_erc.py`).

**Live recommendation:** keep inverse-vol allocator for now. ERC weights
can be computed at audit-time and passed as STARTING WEIGHTS into the
allocator (which then adjusts dynamically based on rolling EWMA vol).
The dynamic vol-targeting in the live allocator is the right operational
model; ERC is the right OFFLINE BENCHMARK for weight checking.

### Decision 3: Strategy registry curation

**Current:** STRATEGY_REGISTRY has 15+ strategies including many that
were V3.6 RETIRED.

**V3.7 action:** Two paths:
- **A. Prune** — remove retired entries to keep the registry clean
- **B. Mark** — add a `v36_status` field (RETIRED, LIVE, CONDITIONAL,
  SHADOW) to each entry; sets filter by status

Recommendation: **A (prune)**. Keeping retired strategies in code is a
landmine. The `Retirement Registry.md` already preserves the lessons.

### Decision 4: Watchdog responsibilities

**Current `watchdog_gem.py`**: restart node on failure, check halt status.

**For V3.7 multi-strategy**:
- Same restart logic (single TradingNode failure → kill + restart all)
- ADD per-strategy parity monitoring (CUSUM drift) — log warnings, not auto-halt
- ADD daily summary write to `.tmp/portfolio_summary.json` for ops UI
- ADD startup pre-flight: assert L65 joint ruin gate passes given current strategy set + weights

## Why this is the right approach

1. **Uses 1070 lines of existing tested code** rather than rebuilding
2. **Multi-strategy from the start** — V3.7 framework's portfolio focus
3. **Future-proof** — adding fx_carry / ic_equity / new strategies = add
   to registry + STRATEGY_SETS entry
4. **Operational simplicity** — one container, one process, one log file
5. **Aligns with V3.7 lessons** — L65 joint, L67 portfolio, ERC, crisis
   stress — all framework primitives expect multi-strategy portfolio
   thinking
6. **Allows future complexity** to be added incrementally (per-strategy
   CUSUM, baseline-class-aware allocator, etc.)
